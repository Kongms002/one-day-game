# 보스 리팩토링 정교 실무 가이드

작성일: 2026-03-08
적용 대상: Unity 프로젝트 보스 전투(실시간 스킬/보스 AI) 전면 정비
참고 기준: SKILL(Unity C#), lifecycle.md, component-checklist.md

이 문서는 기존 설계를 유지하면서 점진적으로 리스크를 줄이는 방식으로, 보스 확장성(스킬 2~3개 이상), 의존성주입, SOLID, OOD를 동시에 만족시키기 위한 실무형 로드맵이다.

## 1) 왜 기존 방식이 깨지기 쉬운가 (현 상태 진단 관점)
- 보스 상태(HP/페이즈/쿨다운), 타겟팅, 투사체 스폰, 데미지 산정이 한 클래스에 섞이면 스킬 추가 시 `if-else` 폭증
- 보스별 고유 수치를 코드 하드코딩하면 밸런스 실험 주기가 느려짐
- 씬 변경/재시작 시 이벤트 구독 정리 누락, FindObjectOfType 잦은 사용으로 런타임 오염
- `Update()` 폴링 기반으로 설계되면 피킹/스킬 실패 케이스가 커도 감지 지연

## 2) 정렬할 4개 축

### 2.1 아키텍처 축 (객체 지향)
- 보스 도메인(상태/페이즈/결정)과 실행(데미지/이펙트/생성)을 분리
- 스킬은 `ITargetingStrategy + ISkillEffect[] + IExecutionPolicy` 조합으로 구성
- 스킬 1개가 여러 동작(투사체, 영역, 데미지, 버프)을 내장 가능

### 2.2 데이터 축 (설정 중심)
- 스킬/페이즈/난이도/가중치 모두 ScriptableObject 기반
- 코드 변경 없이 보스별 스킬 조합, 순위, 가중치 변경 가능

### 2.3 의존성 축 (DI + DIP)
- `MonoBehaviour`는 생성/바인딩 역할, 비즈니스 로직은 인터페이스 주입 서비스
- 런타임 참조 검색은 최소화
- 테스트 가능한 계층 구조로 분리

### 2.4 운영 축 (검증/유지보수)
- 보스 실행 로그와 KPI를 기준으로 데이터 수정
- 유효성 검사(스키마) + 이벤트 중심 디버그로 운영 이슈 즉시 추적

## 4) 핵심 인터페이스(권장 시그니처)

```csharp
public interface IBossSkill
{
    string Id { get; }
    float Cooldown { get; }
    float Weight { get; }
    Task ExecuteAsync(BossSkillContext ctx, CancellationToken cancellationToken);
}

public interface ITargetingStrategy
{
    IReadOnlyList<Target> SelectTargets(BossSkillContext ctx);
}

public interface ISkillEffect
{
    Task ExecuteAsync(BossSkillContext ctx, IReadOnlyList<Target> targets, CancellationToken cancellationToken);
}

public interface IDamageResolver
{
    DamageResult Calculate(BossDamageInput input);
}

public interface IProjectileSpawner
{
    void SpawnProjectile(ProjectileSpec spec, BossSkillContext ctx);
}

public interface IAreaResolver
{
    IReadOnlyList<Collider2D> ResolveArea(AreaSpec spec, Vector2 center);
}

public interface IBossPhaseEvaluator
{
    BossPhase Resolve(BossRunContext runContext, IReadOnlyCollection<BossPhaseState> states);
}

public interface ISkillSelector
{
    BossSkillSO? Select(BossRunContext ctx, IEnumerable<BossSkillSO> candidates);
}
```

`Domain`은 순수 타입만, `Infrastructure`가 Unity 의존 포함 가능, `Presentation`이 시각/애니메이션만 담당.

## 5) 라이프사이클 매핑 (component-checklist + lifecycle 반영)

### 5.1 Awake
- 컴포넌트 캐싱 (`GetComponent`, 자식 탐색) 1회 수행
- 내부 상태 초기화
- DI 컨테이너에서 주입받은 참조를 보관(Find 금지)

### 5.2 OnEnable / OnDisable
- 이벤트 구독/해제
- Pool에서 활성화되는 경우 상태 Reset
- `BossController`는 구독만, 실행 로직은 서비스 위임

### 5.3 Start
- 외부 의존성(씬 오브젝트) 연결이 필요한 경우 최소한 수행
- 시작 시점 스킬/페이즈 초기 바인딩

### 5.4 Update / FixedUpdate
- 입력/타이머/상태 체크는 `Update`에서 이벤트 또는 타이머 기반으로 제한
- 실제 물리 반응이 있는 이동, 푸시 등은 `FixedUpdate` 또는 리지드바디 API 사용

### 5.5 OnDestroy / OnDisable
- 코루틴 중단, 취소 토큰 정리
- 외부 정적 이벤트 구독 해제
- `CancellationTokenSource` 해제

## 6) 보스 리팩토링 단계 (실무형 4주)

### Week 1: 경계 정의 + 안전망 구축
**목표:** 리팩토링 범위 고정 + 하위 호환성 유지
- `BossController` 책임 목록 분해(타이밍, 데미지, 투사체, 페이즈)
- `BossDomain`, `BossService`, `BossData` 경계 생성
- 기존 이벤트/액션 호출 지점 로그화 (현재 동작 보존)
- DI/서비스 위치 결정 (VContainer 또는 Zenject 1종 선택)
- `BossSkillContext`, `BossRunContext`, `DamageResult` 기본 타입 추가

**산출물:** 책임 지도, 서비스 바인딩 계획서, 회귀 지표 기준치

### Week 2: 스킬 파이프라인 마이그레이션 (1개 보스)
**목표:** 스킬 1개 -> 파이프라인 기반 실행
- `BossSkillSO`, `SkillEffectSO` 추가
- `ITargeting + ISkillEffect[]` 구성으로 스킬 1개를 재작성
- 기존 동작과 비교 회귀(데미지 총합, 쿨다운 간격, 실패률)
- `OnEnable/OnDisable` 이벤트 정리 코드 추가

**산출물:** `BossSkillExecutor`, 스킬 파이프라인 PoC, 회귀 리포트

### Week 3: 스킬 2~3개 스케일아웃 + 페이즈 연동
**목표:** 보스마다 최소 2~3개 스킬 조합
- 페이즈별 가중치/블랙리스트 적용
- 투사체형 + 영역형 + 상태이상형 조합 스킬 2종 이상 구현
- 중단 정책(`IsInterrupted`, `IsCancelable`) 및 쿨다운 병목 정책 정의
- 보스 사망/라운드전환 시 실행 중 코루틴 정리

**산출물:** `BossPhaseSO`, `BossSkillSelector`, `BossCooldownService`

### Week 4: 전사양 정제 + 운영화
**목표:** 운영 가능한 문서/툴화
- SO 유효성 검사기 추가 (`NullRef`, 범위, 중복 ID)
- `BossDebugOverlay` 이벤트 로그 (스킬ID, 대상수, 실패사유)
- `Component checklist` 반영한 점검표 적용 (캐시, null-safe, 구독 해제, Update 비용)
- 2개 보스 이상 마이그레이션 완료

**산출물:** QA 체크리스트, 튜닝 가이드, 확장 템플릿

## 7) ScriptableObject 스키마(정교 버전)

### BossConfigSO
- `bossId`, `displayName`, `prefab`, `maxHP`, `moveSpeed`, `baseDamage`, `attackRange`
- `phases`: `BossPhaseSO[]`
- `skillSet`: `BossSkillSetSO`
- `resistanceTable`, `weaknessTable`, `rewardProfile`

### BossPhaseSO
- `phaseIndex`, `hpFrom`, `hpTo`
- `aiSpeedMultiplier`, `cooldownMultiplier`
- `allowedSkillIds`, `forbiddenSkillIds`
- `enter/exitSfx`
- `phaseTransitionActions`

### BossSkillSO
- `skillId`, `weight`, `castTime`, `cooldown`, `isCancelable`, `canQueue`, `priority`
- `targeting` : targeting strategy asset
- `effects`: EffectSO 배열

### SkillEffectSO 하위
- `DamageEffectSO`: baseDamage, multiplier, applyType
- `ProjectileEffectSO`: projectileId/pool/ref, count, spread, speed, lifeSeconds
- `AreaEffectSO`: shape, radius/size, tickInterval, duration, maxTargets
- `StatusEffectSO`: stun/slow/poison/burn 등
- `SummonEffectSO`: prefab, spawnCount, targetMode, lifeSeconds

## 8) 이벤트/통신 방식 (권장)
- 보스 내부는 직접 UI 참조 X
- 보스 이벤트 (`BossSkillStarted`, `BossSkillHit`, `BossSkillCancelled`)는 이벤트 버스로 브로드캐스트
- UI/오디오/카메라는 리스너만 구독
- ScriptableObject Event를 활용해 씬 간 결합 최소화 가능

## 9) SOLID 체크포인트(코드 리뷰용)
- SRP
  - `BossController`는 상태전환/스케줄링만 수행
  - 데미지 계산/이펙트 실행은 별도 서비스
- OCP
  - 새 스킬 효과 추가는 `ISkillEffect` 구현 추가
- LSP
  - 타겟팅 전략/이펙트 구현이 반환 규약 일관적으로 유지
- ISP
  - 인터페이스는 사용 범위별 최소 집합
- DIP
  - 상위 정책은 `IProjectileSpawner`, `IDamageResolver` 등에 의존

## 10) 성능·안전성 제약 (필수)
- `Update` 폴링 최소화: 타이머는 `ITimeService` 기반 갱신 후 단일 체크
- `FindObjectOfType` 제거(초기 세팅 단계 제외)
- 대상 검색 캐시 + 빈번 호출 함수 풀기
- 오브젝트 풀 사용 (투사체/영역 이펙트)
- null-safe 처리: 파괴된 Unity 오브젝트에 대한 방어(`if (!_target) return;` + 소멸 감지)
- 이벤트 중복 구독 방지: OnEnable/OnDisable 균형 필수

## 11) 예시 코드 템플릿(실전형)

### BossController 스켈레톤
```csharp
public sealed class BossController : MonoBehaviour
{
    [SerializeField] private BossConfigSO _config;

    private IBossSkillSelector _skillSelector;
    private IBossPhaseEvaluator _phaseEvaluator;
    private IBossCooldownService _cooldownService;
    private BossRunContext _ctx;
    private CancellationTokenSource _cts;

    private void Awake()
    {
        // 주입은 빌더/DI에서 수행(필요 시 [Inject])
        _ctx = new BossRunContext(_config);
        _cts = new CancellationTokenSource();
    }

    private void OnEnable()
    {
        // 이벤트 구독
    }

    private void OnDisable()
    {
        _cts.Cancel();
        _cts.Dispose();
        // 구독 해제
    }

    private async void Update()
    {
        // 핵심 로직을 스케줄러로 위임
        if (!_ctx.CanAct) return;
        var phase = _phaseEvaluator.Resolve(_ctx, _ctx.Phases);
        var skill = _skillSelector.Select(_ctx, phase.AvailableSkills);
        if (skill != null) await skill.ExecuteAsync(new BossSkillContext(_ctx), _cts.Token);
    }
}
```

### Effect 파이프라인 스켈레톤
```csharp
public sealed class BossSkillExecutor : IBossSkill
{
    private readonly IReadOnlyList<ISkillEffect> _effects;
    private readonly ITargetingStrategy _targeting;

    public async Task ExecuteAsync(BossSkillContext ctx, CancellationToken ct)
    {
        var targets = _targeting.SelectTargets(ctx);
        foreach (var effect in _effects)
            await effect.ExecuteAsync(ctx, targets, ct);
    }
}
```

## 12) Unity 컴포넌트 리뷰 체크리스트(보스 전용)
- [ ] [SerializeField] private + `_` private field 규칙 준수
- [ ] 캐시되지 않은 GetComponent 호출 없음(특히 Update/FixedUpdate)
- [ ] 필수 참조 null 검사(누락 시 disable/로그)
- [ ] OnEnable/OnDisable 구독 대칭
- [ ] Update/FixedUpdate에서 heavy alloc 없음
- [ ] 타이밍/쿨다운은 중앙 타이머에서 일괄 처리
- [ ] ScriptableObject 기준치 유효성 검사
- [ ] OnDestroy에서 코루틴/구독 정리

## 13) QA 시나리오
- 목표: 목표 대상 미존재, 파괴 직후 참조, 씬 전환, 라운드 재시작, 보스 사망 직후에 스킬 실행 중단
- 검증 항목
  - 타겟 미탐색 상태에서 스킬이 예외 없이 실패 처리되는가
  - 페이즈 전환 중 중간 스킬이 중복 실행되지 않는가
  - 쿨다운/가중치가 기대치 대비 정상인지
  - 재시작 후 이전 보스 코루틴이 남아 있지 않은가
