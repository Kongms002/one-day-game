# 보스 리팩토링 비전 및 핵심 원칙 (2026-03-08)

## 목적
기획 의도(보스 스킬 2~3개 이상, 투사체/영역/데미지 확장)를 기준으로, 현재 프로젝트의 전투 시스템을 "확장 가능한 보스 스킬 아키텍처"로 전환한다.

- 보스별 패턴을 코드 수정 없이 데이터로 조합 가능
- 테스트 가능한 책임 단위로 분해
- 의존성 주입(DI)과 인터페이스 중심 설계로 결합도 낮춤
- 장기적으로 10구간 보스 패턴 확장 및 밸런스 운영 비용 최소화

## 기존 구조 문제(리팩토링 타겟)
- 보스 스킬이 단일 클래스/매직 넘버 중심으로 고정되면 신규 스킬 추가 시 핵심 로직까지 수정
- 보스 로직 안에 데미지 계산/타겟팅/이펙트 생성/쿨다운/단계 전환이 혼재
- 설정 데이터가 하드코딩되면 빌드 후 튜닝 피드백 루프가 느려짐
- 씬/런타임 라이프사이클 의존성이 강해 보스만 교체해도 영향 범위가 큼

## 목표 상태(운영 가능한 형태)
- `Boss` 자체는 "상태 머신 + 선택기" 역할만 담당
- 스킬 실행은 `SkillPipeline` 형태로 조합
- 스킬 구성요소를 `ScriptableObject`로 분리하고 빌드/밸런스는 데이터만 수정
- 무기/적/보스 시스템과 동일 규칙의 도메인 서비스로 통일된 DI 구성

## 적용 원칙(코어)
### 1) SRP + 데이터/실행 분리
- `BossEntity`는 행동 의사결정만, 계산/이펙트/판정은 별도 서비스로 분리
- 보스 스탯, 패턴, 페이즈, 룰만 데이터화

### 2) OCP + 컴포저블 패턴
- 보스 스킬은 3단계 조합 가능
  1. 타깃팅 전략(`ITargetingStrategy`)
  2. 스킬 이펙트(`ISkillEffect` 리스트)
  3. 후처리(`Cooldown`, `PhaseTransition`, `Aura` 등)
- 새 스킬은 기존 클래스 수정 없이 조합 클래스/SO 추가로 확장

### 3) DI 우선
- `MonoBehaviour`가 직접 new/newed dependencies를 만들지 않음
- DI 컨테이너가 런타임 객체(서비스, 팩토리, 풀) 조립
- 테스트/에디터 시뮬레이션에서 동일한 컨트랙트(Mock) 사용

### 4) SOLID 정합성
- SRP: 보스 패턴 결정, 타이머/쿨다운, 데미지 계산, 이펙트 생성 분리
- OCP: 스킬 액션은 플러그형 구조로 추가
- LSP: `ISkillEffect`의 모든 구현체는 동일 계약 이행
- ISP: 입력/타깃팅/데미지/이펙트 인터페이스 분리
- DIP: 상위 정책(보스 AI)이 하위 구현(스킬 실행기) 직접 의존하지 않음

### 5) 게임 운영 친화
- 튜닝은 `SO` 수정/주소형 에셋 교체로 처리
- 수치 변경이 코드 배포 없이 hotfix 수준으로 가능하도록 단방향 파이프라인 유지
- 보스 밸런싱 로그(스킬 적중률, 실패율, 회피율)로 운영 지표 수집을 염두에 둠
