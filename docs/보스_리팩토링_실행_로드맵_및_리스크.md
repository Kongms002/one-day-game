## 0) 범위
- 대상: 보스 AI, 스킬 실행, 데미지/투사체/영역 판정, 보스 페이즈/스폰 흐름
- 제외: 기존 무기 시스템 전체 교체(단, 공통 인프라는 공유)

## 1) 현재 구조 정리 (준비 단계)
### 산출물
- 보스가 수행하는 책임 목록(현재 클래스 기준) 추출
- `보스 상태 변경 지점` 맵핑: 스폰, 페이즈 전환, 사망, 스킬 발동
- 외부 의존성 목록: `MonoBehaviour`, 인스펙터 참조, 시퀀스/코루틴, 풀링, 이벤트 버스

### 체크
- 스킬 동작 중 타겟 탐색, 데미지, 투사체 스폰, 이펙트 실행이 하나의 코드블록 안에 있는지 확인
- 하드코딩된 수치(쿨다운, 데미지, 반경)가 있는지 검색

## 2) 인터페이스 우선 계약 정리
### 새 인터페이스 제안(예시)
- `IBossSkill`
- `IBossPhaseEvaluator`
- `ISkillSelector`
- `ITargetingStrategy`
- `ISkillEffect`
- `IDamageCalculator`
- `IProjectileSpawner`
- `IAreaResolver`
- `ICooldownScheduler`
- `IBossEventDispatcher`

### 작업 방식
- 각 인터페이스를 먼저 정의하고 기존 구현은 어댑터로 래핑
- 서비스 호출부는 인터페이스 기반으로 교체, 구현은 점진적으로 분리

## 3) 도메인 분리 (1차 스프린트)
### 목표(필수)
- `BossController`에서 아래 책임 제거
  - 데미지 수치 산정
  - 투사체 생성 규칙
  - 타겟 검색 규칙
  - 영역 계산/필터링
  - 페이즈별 스킬 가중치 정책

### 새 컴포넌트
- `BossRunContext`, `BossPhase`, `BossSkillState`
- `BossAIStateMachine`(Idle/Combat/Recover/Dead)
- `BossSkillSelector`
- `BossCooldownTracker`

## 4) 스킬 파이프라인 적용 (2차 스프린트)
### 목표
- 보스 1개 스킬이 최소 3개 이펙트를 가질 수 있도록 `SkillPipeline` 실행 구조 정착
  - `Targeting -> Damage -> Projectile/Area -> PostProcess`

### 작업
- SkillSO + EffectSO 추가
- `BossSkillExecutor` 구현
- 기존 단일 스킬 루틴 1개를 기준으로 파이프라인 전환 후 동작 동일성 확인
- 이후 동일 패턴을 2개 이상의 스킬로 확장

## 5) DI 정착 (3차 스프린트)
### 권장 컨테이너
- VContainer 또는 Zenject(Extenject) 중 기존 프로젝트 호환성 고려

### 바인딩 정책
- 인프라 서비스: `ProjectilePool`, `FxFactory`, `RandomProvider`
- 핵심 서비스: `IDamageCalculator`, `IAreaResolver`, `IProjectileSpawner`, `ICooldownScheduler`, `IBossEventDispatcher`
- 보스별 정책: `IPhaseEvaluator`, `ISkillSelector`, `IBossSkillCatalog`

### 주의점
- `MonoBehaviour`의 `Start/Awake`에서 `FindObjectOfType` 의존성 축소
- 시리얼라이즈된 참조는 최소 유지하고 런타임 조립은 DI로 수렴

## 6) 보스 확장 패턴 (3~5개 마이그레이션)
### 단계
1. 기존 보스 중 하나를 `BossConfigSO + BossPhaseSO + BossSkillSO`로 마이그레이션
2. 스킬 2종 구성 추가(기존 1스킬에서 2~3개 운영)
3. 투사체 + 영역 혼합 스킬 1개 정의
4. 동일 패턴 템플릿으로 나머지 보스 점진 변환

## 7) 성능·안정성 개선
- 스킬 스캔 캐시: 매 프레임 SO 재탐색 금지
- 타겟링 전략은 레이어 기반 precomputed 컬렉션 사용
- 투사체는 풀링 강제 사용(생성/파괴 금지)
- `DamageResult` 계산은 값타입 + 캐시로 구성해 할당 최소화
- 보스 페이즈 전환 시 실행 중 코루틴 정리 루틴 강화

## 8) 위험 및 대응
### R1: 스킬 파이프라인 과다 분리로 디버그 비용 증가
- 대응: `BossDebugOverlay`, 실행 로그(스킬ID, 타깃, 데미지, 실패이유)

### R2: SO 파라미터 튜닝 실수로 런타임 크래시
- 대응: 런타임 유효성 검사 validator + 에디터 검증 툴

### R3: DI 적용 초기엔 오버헤드 증가
- 대응: 컨테이너 의존성은 게임 진입 시 1회 바인딩, 프레임 패스에서 resolve 호출 최소화

### R4: 페이즈 전환 동기화 버그
- 대응: `PhaseTransitionEvent`를 통해 단일 진입 지점에서 상태 변경 통제

## 9) 검증 KPI(배포 전)
- 보스 스킬 추가 시간: 기존 대비 70% 단축 목표
- 기존 보스 회귀: 데미지 총합 ±5% 이내
- 스킬 생성 누락률: 0%
- 3개 스킬 이상 동시 구성 보스의 런타임 크래시 0건
- 1주간 10회 난이도 테스트에서 프레임 드롭 5% 이하
