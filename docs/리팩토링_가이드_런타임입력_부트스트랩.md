# 런타임 입력 오케스트레이션 리팩토링 가이드

## 목적

- `GameBootstrap`의 입력 초기화/활성/위치 조정 책임을 줄여 모듈화를 강화한다.
- 기존 동작(시작/씬 전환/재시작/게임 종료 시 입력 상태)은 유지한다.
- 변경 범위를 `RuntimeInputPort`와의 상호작용 중심으로 축소한다.
- 씬 전환 시 입력 포트 교체가 발생해도 이벤트 구독과 플레이어 입력 참조가 끊기지 않도록 재바인딩 안전성을 확보한다.

## 반영 대상 코드(핵심)

1. `Assets/Scripts/Presentation/Bootstrap/GameBootstrap.cs`
2. `Assets/Scripts/Presentation/Input/RuntimeInputPort.cs` (참조 포인트 확인용)

현재 `GameBootstrap`에서 입력 관련 메서드로 직접 관리되는 구간
- `EnsureRuntimeReferences()`
- `SetRuntimeInputEnabled(bool)`
- `ShouldRuntimeInputBeEnabled()`
- `RepositionRuntimeJoystickAtPlayerStart(Vector3)`

이 4개를 새로운 오케스트레이터로 이동하는 방식으로 진행한다.

## 권장 구조

- `Assets/Scripts/Presentation/Bootstrap/RuntimeInputCoordinator.cs` 신규 추가
  - 클래스: `RuntimeInputCoordinator`
  - 책임:
    - 런타임 입력 오브젝트/포트 보장
    - 입력 활성/비활성 제어
    - 입력 가능 여부 계산(런/죽음 상태 반영)
    - 조이스틱 시작 위치 재배치
    - 불필요한 레거시 오버레이 종료 버튼/시작 UI 숨김
    - 입력 포트 교체 시 구독 경계를 안전하게 재연결할 수 있는 오케스트레이션 제공

`RuntimeInputCoordinator`는 Unity 의존이 있지만 `GameBootstrap`은 직접 오케스트레이션을 하지 않도록 한다.

### 객체지향/유니티 보강 원칙

- SRP: 입력 오케스트레이션은 코디네이터에 모으고, `GameBootstrap`은 런 조립과 라이프사이클 제어만 담당한다.
- DIP: 런타임 포트 구체 타입 접근은 코디네이터로 숨기고, 부트스트랩은 코디네이터 API 중심으로 의존한다.
- Lifecycle: 씬 로드마다 포트 재획득 + 기존 구독 해제 + 신규 구독 연결 + 플레이어 포트 재바인딩을 한 경로에서 처리한다.
- Defensive: Unity 객체 접근은 명시적 null 가드로 처리하고, 캐스팅 실패는 no-op으로 종료한다.

## 적용 단계

### 1) 가이드 전제 점검 (Readiness)

- `GameBootstrap` 내부에서 다음 호출부 확인
  - `Awake()`에서 입력 초기화 순서
  - `OnActiveSceneChanged()`에서 상태 동기화
  - `OnRunEnded()` / `OnRunRestarted()`에서 토글 동작
  - 플레이어 초기 배치 후 조이스틱 재배치 호출
- `RuntimeInputPort`는 `IInputPort`로 주입되어 있으나, 실제 동작은 런타임 타입 캐스트(`RuntimeInputPort`)가 필요함을 확인.
- `_runSession`의 생명주기와 `IsDead` 상태가 입력 상태 판단 조건에 들어감.

### 2) 신규 코디네이터 추가

`RuntimeInputCoordinator`는 다음 공개 API를 갖는다.

- `IInputPort EnsureRuntimeReferences(IInputPort inputPort, bool useUltimateInput, string actionMapName, string ultimateActionName)`
  - 기본 동작:
    - `HideLegacyStartOverlay()` 수행
    - `RuntimeInputPort` 탐색/생성
    - 비활성 오브젝트면 활성화 복원
    - `AutoBindFallbackJoysticks()` 호출
    - `ConfigureUltimateInput(...)` 및 초기 입력 상태 계산 반영
- `void SetRuntimeInputEnabled(IInputPort inputPort, bool enabled)`
  - `RuntimeInputPort.SetInputEnabled(enabled)` 위임
- `bool ShouldRuntimeInputBeEnabled(bool isRunInputEnabled, RunSessionService runSession)`
  - 조건: `isRunInputEnabled && runSession != null && !runSession.IsDead`
- `void RepositionRuntimeJoystickAtPlayerStart(IInputPort inputPort, Vector3 playerStartPosition)`
  - `RuntimeInputPort.TrySetJoystickToWorldPosition(...)` 위임
- `private static void HideLegacyStartOverlay()`
  - `TapToStart`, `Title` 게임오브젝트 비활성화

> 구현은 기존 `GameBootstrap`의 현재 동작을 1:1로 옮기는 방식으로 한다.
> 상태 계산 메서드는 입력 파라미터만 사용하는 순수 함수 형태를 유지한다.

### 3) GameBootstrap 리팩토링

1. 필드 추가
   - `_runtimeInputCoordinator` 인스턴스 생성

2. 호출부 교체
   - `Awake()`:
     - 기존 `EnsureRuntimeReferences();`
     - 기존 `SetRuntimeInputEnabled(false);`
     - -> 코디네이터 사용으로 교체
   - `OnActiveSceneChanged()`:
     - 기존 보장/토글 호출 두 개를 코디네이터 호출로 교체
     - 입력 포트가 교체될 경우 기존 `FrameTick` 구독 해제 + 신규 포트 구독 재연결
     - `PlayerView.BindInputPort(...)`로 플레이어 입력 포트 참조 갱신
   - `OnRunEnded()` / `OnRunRestarted()`:
     - 입력 토글을 `RuntimeInputCoordinator.Should...` + `SetRuntimeInputEnabled` 체인으로 교체
   - `Awake()`, `OnRunRestarted()` 플레이어 시작 위치 복귀 직후 조이스틱 이동도 코디네이터 경유

3. `GameBootstrap` 정리
   - 기존 4개 메서드(`EnsureRuntimeReferences`, `SetRuntimeInputEnabled`, `ShouldRuntimeInputBeEnabled`, `RepositionRuntimeJoystickAtPlayerStart`) 제거
   - 삭제한 메서드 대신 코디네이터 호출만 남기고, 시그니처가 줄어들었는지 확인

### 4) 안전성 가드 (필수)

- 코디네이터는 `IInputPort`가 런타임 타입이 아닌 경우 `as RuntimeInputPort` 캐스팅 실패 처리로 조용히 no-op 처리.
- `_runSession` 미초기화 구간(`Awake` 초반)에서는 입력 활성화가 false가 되도록 계산이 유지되는지 확인.
- `EnsureRuntimeReferences` 직후 `SetRuntimeInputEnabled`에서 상태가 중복 계산되어도 기존 동작과 동일해야 함.
- 씬 전환 이후 포트 인스턴스 변경 시 이벤트 구독이 한 번만 유지되는지(`-=` 후 `+=`) 확인.
- Unity 오브젝트는 `?.`에 의존하지 않고 명시적으로 null 가드를 적용.

### 5) 검증 체크리스트

1. `Awake`에서 시작 시
   - 입력이 비활성 상태로 시작
   - 런타임 포트/조이스틱이 존재
   - TapToStart / Title 오브젝트가 숨겨짐

2. 실제 조작 시작 시점
   - `SetRuntimeInputEnabled(true)` 호출 후 입력이 정상 반영

3. 씬 변경
   - `OnActiveSceneChanged`에서 입력 오케스트레이션이 재동기화되고 조이스틱이 시작 위치로 이동
   - 포트 교체 상황에서도 `OnInputFrame` 이벤트 수신이 유지됨

4. 라운드 종료/재시작
   - 종료 시 조이스틱 입력 꺼짐
   - 재시작 시 입력 복원 + 플레이어 위치/조이스틱 위치 재설정

5. 회귀 위험 점검
   - `UltimatePressButton` 비활성 동작 유지
   - 조이스틱 자동 생성/바인딩 경로가 깨지지 않았는지 확인

## 예상 산출물

- `RuntimeInputCoordinator.cs` 1개 신규 파일
- `GameBootstrap.cs` 입력 책임 축소(메서드 4개 이관)
- `PlayerView.cs` 입력 포트 재바인딩 메서드 추가(`BindInputPort`)
- 동작 변경 없음, 결합도 감소

## 참고 가이드(리뷰 항목)

코드 리뷰 시 아래 항목을 체크한다.
- 입력 토글 계산식이 오직 `Run` 상태 플래그(`_isRunInputEnabled`)와 `RunSessionService.IsDead`만 바뀐 의도인지
- `_inputPort` null 방어가 충분한지
- `RuntimeInputPort` 특화 API 접근이 필요한 부분을 안전하게 캡슐화했는지
