적용 범위: `Assets/Scripts` 전체 (Bootstrap, Input, Player, Weapon, Boss, Enemy, UI, Repository, Policies)

## 2) 현재 코드 진단(요약)

### 강점
- `Domain`/`Application`/`Infrastructure`/`Presentation` 폴더 구조 존재
- 보스 스킬 파이프라인은 SO 기반 구성의 골격이 이미 존재
  - `BossConfigSo.cs`, `BossSkillRuntimeServices.cs`, `BossSkillBrain.cs`
- 스폰/정책/난이도 계층이 분리됨
  - `SpawnService.cs`, `DefaultSpawnPolicy.cs`, `DefaultDifficultyPolicy.cs`
- 입력 인터페이스(`IInputPort`)와 런타임 입력 오브젝트 분리
  - `RuntimeInputPort.cs`, `RuntimeInputCoordinator.cs`

### 리스크/개선 포인트
- `GameBootstrap.cs`가 너무 많은 책임을 가짐(생성, 초기화, UI 바인딩, 스폰 관리, 입력 제어, 씬 처리)
- 플레이어/무기/공격 판정이 `PlayerView.cs`에서 묶여있어 테스트/확장 난이도 상승
- `PlayerController.cs`(레거시) 유지로 경로 분기/중복 위험
- `GameManager.cs`와 Bootstrap 간 초기화 경합 가능성
- `Object.Find*`, `GameObject.Find` 의존이 일부 런타임 경로에 존재
  - `RuntimeInputCoordinator`, `GameBootstrap`, `GameManager`

## 3) 리팩토링 원칙 (SKILL + Lifecycle + Component-checklist 반영)

### 3-1) Lifecycle 규칙
- `Awake`: 컴포넌트 캐시, 내부 상태 초기화
- `OnEnable/OnDisable`: 이벤트 구독/해제
- `Start`: 다른 객체 의존이 확정된 뒤(부트스트랩 조립 후) 초기 동기화
- `Update`: 상태 전이 트리거/경량 체크만 수행
- `OnDestroy`: 구독 정리, 코루틴/타이머 취소

### 3-2) 컴포넌트 규칙
- `[SerializeField] private` 사용, public field 지양
- `[RequireComponent]`, `Tooltip`, `Header`, `Range` 적극 적용
- OnEnable/OnDisable에서 구독 대칭
- Update/FixedUpdate에서 `GetComponent` 반복 호출 제거

### 3-3) SOLID 정합
- SRP: 각 클래스는 “하나의 역할” 원칙
- OCP: 스킬/무기/스폰 효과 추가 시 클래스 추가 + 등록으로 대응
- LSP/ISP: 인터페이스는 짧고 목적 단일
- DIP: `new` 직접 생성 최소화, 생성은 조립기/Factory에서

## 4) 목표 아키텍처(최종 상태)

### 4-1) Layer 분리안

```text
Bootstrap (Orchestration)
  GameCompositionRoot
  SessionCoordinator
  SceneCoordinator
  InputCoordinator
  Infrastructure binding/adapter

Application
  RunSessionService (Domain state)
  SpawnService + SpawnRequestHandler
  WeaponOrchestrator + WeaponLoadoutService
  Boss systems(요약: SkillCatalog, Selector, Executor, Cooldown, PhaseResolver)
  UpgradeService

Presentation
  PlayerControllerFacade
  EnemyPresenter
  HudPresenter
  MapPresenter
  UI Command Handlers
  Effects visual layer

Domain
  Value objects / policies interfaces / snapshots / contracts
  No Unity dependency

Infrastructure
  UnityRandomService, PlayerPrefsRunRepository
  Factories + Pool registries
  SO builders/readers
```

## 5) 전체 프로젝트 모듈별 정교 리팩토링 플랜

### A. Bootstrap/런타임 조립(최우선)
**목표:** 런타임 부트스트랩 고립

#### 현재 대상 파일
- `Assets/Scripts/Presentation/Bootstrap/GameBootstrap.cs`
- `Assets/Scripts/Presentation/Bootstrap/RuntimeInputCoordinator.cs`
- `Assets/Scripts/GameManager.cs`

#### 단계
1. `GameCompositionRoot` 추가
   - 서비스 생성/의존성 주입/초기화 순서를 한 곳으로 모음
2. `GameBootstrap`을 `SessionCoordinator`로 축소
   - 부트스트랩은 오직 `Initialize()`, `OnRunStart()`, `OnRunEnd()`, `OnSceneChanged()` 바인딩만 담당
3. `GameManager`와 `GameBootstrap`의 역할 분리
   - `GameManager`는 scene bootstrap 트리거만 보유
   - 부트스트랩 생성 및 정책 주입은 `GameCompositionRoot`로 이동
4. `OnEnable/OnDisable`의 구독/구독해제 대칭성 확보
5. `ScanMissingScriptsInActiveScene` 류 진단 코드 분리 (개발자 모드 only)

### B. 입력/액션 파이프라인
**목표:** 입력 타입 불변화 + 프레임 바인딩 통제

#### 대상
- `Assets/Scripts/Domain/Input/IInputPort.cs`
- `Assets/Scripts/Presentation/Input/RuntimeInputPort.cs`
- `Assets/Scripts/Presentation/Bootstrap/RuntimeInputCoordinator.cs`
- `Assets/Scripts/Presentation/Gameplay/PlayerView.cs`

#### 단계
1. `InputFrameBus` 추가(프레임 단위 입력 버퍼)
2. `SetInputAxis`와 조이스틱 상태 업데이트를 오직 버스 구독자가 수행
3. `RuntimeInputCoordinator`에 초기화/재인식 정책만 두고, 입력 포맷 변경은 `RuntimeInputPort`에서 캡슐화
4. `GameBootstrap`에서 `FindObjects` 방식 최소화 (RuntimeInputPort 주입)
5. 조이스틱 생성/재위치 로직 정리(테스트 가능한 정책 메서드로 분리)

### C. 게임 루프/상태/이벤트
**목표:** `Update` 통합 및 이벤트 드리븐 운영

#### 대상
- `RunSessionService.cs`
- `SpawnService.cs`
- `GameBootstrap.cs`
- `GameHudPresenter.cs`

#### 단계
1. `GameLoopCoordinator`를 추가해 `RunSessionService.Tick`, `SpawnService.Tick`, `LevelProgress`, UI 갱신을 1회 관제
2. `RunState` 이벤트(`SnapshotChanged`, `RunEnded`, `Restarted`) 소비를 UI, 스폰, 보스, 입력 게이트에서 명확히 분리
3. 시작/종료/재시작 시나리오를 상태 머신으로 정리:
   - `Idle` → `Running` → `Dead` → `Result` → `Restarting`
4. `Time.timeScale` 변화/재시작/라운드 정지 정책을 하나의 라인에서 관리

## 8) 완료 기준(DoD)
- 모든 핵심 시스템이 단위 이벤트 1곳에서 구독/해제
- `GameBootstrap`의 책임이 1차 분해되고, 재시작/씬전환/죽음 상태에서 입력/쿨다운/코루틴 정합성 확보
- 보스가 스킬 2~3개 조합에서 투사체/영역/딜레이 패턴 안정 실행
- 무기 슬롯 4개가 정책 기반 업그레이드와 호환
- 적 생성(일반/보스)과 아이템 생성이 SpawnService 이벤트로 일괄 처리
- UI는 `IRunState`/`IWeaponLoadoutReadModel` 변경 이벤트만으로 동기화
